AIを使い始めると、最初は「質問すると答えてくれる道具」として便利に感じます。文章のたたき台、調査の整理、アイデア出し、メール文面の修正など、すぐに効果が出る使い方は多くあります。

ただし、そこで止まると毎回同じ相談を繰り返すことになります。前回うまくいった指示、判断基準、成果物、修正内容が残らず、AI活用がその場限りの会話になります。

AI業務改善ラボでは、AIを「相談相手」ではなく「業務改善の実行基盤」として扱います。重要なのは、AIの回答を使い捨てにしないことです。

## 相談、実行、保存、再利用を分ける

AI活用を業務改善に変えるには、作業を4つに分けます。

1. 相談する
2. 実行する
3. 保存する
4. 再利用する

相談は、ChatGPTやClaudeのような会話AIが得意です。曖昧な課題を整理し、作業方針を決め、プロンプトやチェックリストのたたき台を作れます。

実行は、CodexやClaude Codeのようにリポジトリやファイルを扱えるAI Agentが得意です。実際にファイルを作り、編集し、buildし、commitまで進められます。

保存は、ObsidianのようなMarkdownベースの知識管理が向いています。AIとのやり取りを、次回も使えるPrompt、Skill、Agent設計、Evalとして残します。

再利用は、人間が確認しながら行います。AIにすべて任せるのではなく、再利用してよい手順、使ってはいけない情報、確認すべき観点を決めます。

## Promptは入力文ではなく作業設計書

Promptは、その場でAIに投げる文章だけではありません。業務改善で使うなら、次の要素を持つ作業設計書として扱います。

- 目的
- 入力する情報
- 禁止事項
- 出力形式
- 判断基準
- 人間が確認する項目

例えば「記事を書いて」では再利用できません。「誰向けに、何を、どの粒度で、どの見出し構成で、どの内部リンクへつなげるか」まで決めると、次回も同じ品質に近づけられます。

Promptの基本は [Prompt / Skill / Agent の入門ページ](/prompt-skill-agent/) と [PromptSkillAgentの違いを実務目線で整理する](/articles/prompt-skill-agent-practical-difference/) で扱います。

## Skillは繰り返す手順を保存するもの

Skillは、何度も使う作業の手順です。単なるメモではなく、AIに作業を任せるときの型として保存します。

Skillに入れるべきものは、作業手順だけではありません。

- いつ使うか
- 何を入力するか
- 何をしてはいけないか
- どう検証するか
- 完了条件は何か

例えば「公開前チェック」というSkillなら、build、リンク確認、OGP確認、robots確認、sitemap確認、秘密情報確認まで含めます。これにより、作業者が変わっても同じ確認を再現できます。

## Agentは判断しながら作業する担当者

Agentは、PromptやSkillよりも実行寄りの概念です。目的、権限、利用できるツール、完了条件を持ち、判断しながら作業を進めます。

ただし、Agentに何でも任せると危険です。本番サーバー接続、DNS変更、外部アカウント操作、顧客情報の扱いなどは、人間の判断と承認が必要です。

このサイトでは、Codexを開発Agentとして扱っています。詳しくは [CodexでAstroサイトを作った流れ](/articles/codex-astro-site-build-log/) と [ChatGPTとCodexの役割分担](/articles/chatgpt-codex-role-split/) にまとめます。

## GuardrailsはAIに任せない範囲を決めること

Guardrailsは、AIの行動範囲を制限するためのルールです。難しい言葉に見えますが、実務では「やってよいこと」と「やってはいけないこと」を明確にするだけでも効果があります。

このサイトの作業では、次のようなGuardrailsを置いています。

- 本番サーバーへ接続しない
- お名前.com管理画面へログインしない
- FTP / SSH / DBへ接続しない
- `.env` や認証情報を作らない
- 顧客情報や個人情報を扱わない
- `dist/` を本番へ自動アップロードしない

AIに任せるほど、禁止事項は具体的に書く必要があります。

## Evalは良し悪しを判断する基準

Evalは、AIの出力を評価するための基準です。実務では、完璧な自動評価よりも、再現できるチェックリストのほうが先に役立ちます。

例えば記事作成なら、次のようなEvalを置けます。

- titleとdescriptionがあるか
- 内部リンクがあるか
- 専門用語を説明しているか
- CTAがあるか
- 誇張しすぎていないか
- buildが通るか

Evalを作ると、AIの出力を感覚で判断しなくて済みます。良かった点と直す点をログに残せば、次のPromptやSkillも改善できます。

## AI秘書兼スタッフOSという考え方

AI秘書兼スタッフOSとは、AIを単体ツールとして使うのではなく、業務の中に配置する考え方です。

ChatGPTは相談役、Codexは作業担当、Obsidianは記憶層、人間は判断者です。それぞれの役割を分けると、AI活用は安定します。

重要なのは、AIにすべてを任せることではありません。AIが得意な作業を任せ、人間が判断すべきところを残し、成果物を保存して再利用することです。
